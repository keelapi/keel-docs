# Errors

Stable minimums to depend on across Keel's public runtime surfaces:

- HTTP status code
- machine-readable code
- human-readable message

Do not assume a single universal error envelope across permit, execution, proxy, and request-level API failures.

## Error shapes by surface

| Surface | Typical result shape | Parse path | Notes |
| --- | --- | --- | --- |
| `POST /v1/permits` deny decision | Permit object, usually not a top-level error object | `decision`, `reason`, `decision_details.deny_code` | Permit denials are decision results, not execution failures, and do not include an `error` object. |
| `POST /v1/executions` denied response | Normalized execution envelope with `status: "denied"` and `error` | `status`, `governance.permit_id`, `error.code`, `error.message` | Denied executions stay inside the normalized execution envelope. |
| `POST /v1/executions` failed response | Normalized execution envelope with `status: "failed"` and `error` | `status`, `governance.permit_id`, `error.code`, `error.message` | Provider or downstream execution failures still return an execution object. |
| `POST /v1/execute` denied or failed response | Normalized execution envelope with `error`, plus `resolved` | `status`, `governance.permit_id`, `error.code`, `error.message`, `resolved` | Same parsing model as `/v1/executions`, with one extra `resolved` block. |
| Transport, framework, or top-level API error | Top-level `{ "error": { ... } }` object | `error.code`, `error.message`, `error.details` | Use this shape for validation, auth, not-found, idempotency-conflict, and similar request-level failures before an execution object exists. |
| `POST /v1/proxy/*` provider proxy error | Provider-native error object | Provider-native path such as `error.message` | Proxy routes do not return the normalized execution envelope. Parse them the same way you would parse the provider directly. |

For the execution routes:

- A denied `/v1/executions` response is a normalized execution envelope with `status: "denied"`.
- A failed `/v1/executions` response is a normalized execution envelope with `status: "failed"`.
- A transport, framework, or top-level API error is a standalone `{ "error": ... }` object because no execution envelope was created.
- A provider proxy error is provider-native, not normalized.
- Permit denials are decision results, not execution failures, and do not include an `error` object.

## Examples by surface

The permit decision examples below reflect the current documented contract for permit decisions.

### `/v1/permits` deny

HTTP status: `200`

```json
{
  "id": "permit_01jyf4p6c3m7n9q2t5v8w1x4y",
  "decision": "deny",
  "reason": "model_not_permitted_for_project",
  "actions": [
    {
      "type": "deny"
    }
  ],
  "metadata": {
    "policy_id": "pol_model_allowlist",
    "policy_version": "2026-03-15",
    "evaluated_at": "2026-03-26T18:43:02Z"
  },
  "decision_details": {
    "outcome": "policy_deny",
    "matched_policies": [
      "pol_model_allowlist"
    ],
    "rule_id": "rule_block_unapproved_models",
    "deny_code": "model_not_allowed"
  }
}
```

Parsing note: Permit denials are decision results, not execution failures, and do not include an `error` object. Read `decision`, `reason` if present, and `decision_details.deny_code`; do not expect `error.code`.

### `/v1/permits` allow

HTTP status: `200`

```json
{
  "id": "permit_01jyf4m3n8q2r6t9v1w5x7y0z",
  "decision": "allow",
  "reason": "request_within_model_and_budget_policy",
  "actions": [
    {
      "type": "allow"
    },
    {
      "type": "report_usage"
    }
  ],
  "constraints": {
    "max_output_tokens": 300,
    "allowed_modalities": ["text"],
    "execution_mode": "sync"
  }
}
```

Parsing note: Allow decisions are permit results too. Parse `decision` and `reason`; do not expect `error.code`. For the fuller allow response, see [Permits](/permits#response-examples).

```json
  },

    "total_ms": 19
  },
  "usage": {
    "input_tokens": 0,

```

Parsing note: parse `status`, `governance.permit_id`, and `error.code`; denial is still an execution envelope.

### `/v1/proxy/openai` provider-style error

HTTP status: `400`

```json
{
  "error": {
    "message": "Unsupported parameter: 'max_completion_tokens'.",
    "type": "invalid_request_error",
    "param": "max_completion_tokens",
    "code": "unsupported_parameter"
  }
}
```

Parsing note: parse this as an OpenAI-style provider error, not as a Keel execution envelope.

### Generic top-level API error object

HTTP status: `409`

```json
{
  "error": {
    "code": "idempotency_conflict",
    "message": "The same idempotency key was already used with a different semantic request.",
    "details": {
      "idempotency_key": "exec-sync-001"
    }
  }
}
```

Parsing note: when the response is top-level `{ "error": ... }`, no normalized execution object was created.

## Status and retry matrix

Usual status means the documented status you should plan for in client logic, not an exhaustive guarantee for every deployment detail.

On `/v1/permits`, parse `decision`, `reason` if present, and `decision_details.deny_code`; do not expect `error.code`.

| HTTP status | Code | Surface(s) | Retryable? | Meaning |
| --- | --- | --- | --- | --- |
| `200` | `model_not_allowed` | Permits deny decision | No | The permit was denied because the requested model is not allowed for the project. |
| `400` | `invalid_request` | Permits, executions, execute, proxy, request-level API | No | The request body, headers, or field combination is invalid. |
| `401` | `unauthorized` | Permits, executions, execute, proxy, request-level API | Conditionally | Missing or invalid credentials. Retry only after fixing auth. |
| `403` | `forbidden` | Permits, request-level API | No | The caller is authenticated but not allowed to use the route or resource. |
| `403` | `insufficient_scope` | Permits, executions, execute, proxy, request-level API | No | The credential does not have the required scope. |
| `403` | `policy_denied` | Executions, execute | No | Keel evaluated the permit seam and denied the request. |
| `403` | `provider_key_not_configured` | Executions, execute, proxy | No | The project does not have the required provider key configured. |
| `403` | `prompt_firewall_blocked` | Executions, execute, proxy | No | Prompt Firewall blocked the request before provider dispatch. |
| `403` | `daily_budget_exceeded` | Executions, execute | No | The request would exceed the project's daily budget. |
| `403` | `monthly_budget_exceeded` | Executions, execute | No | The request would exceed the project's monthly budget. |
| `403` | `request_cost_exceeded` | Executions, execute | No | The single request exceeds the allowed cost ceiling. |
| `403` | `pricing_not_configured` | Executions, execute | No | Keel cannot evaluate required pricing data for the request or routing policy. |
| `403` | `budget_guardrail_daily_spike_exceeded` | Executions, execute | No | A daily spike guardrail blocked the request. |
| `403` | `budget_guardrail_monthly_threshold_exceeded` | Executions, execute | No | A monthly threshold guardrail blocked the request. |
| `403` | `budget_guardrail_rate_exceeded` | Executions, execute | No | A rate guardrail was exceeded and the route failed closed. |
| `429` | `budget_guardrail_rate_throttled` | Executions, execute | Yes | A rate guardrail throttled the request. Retry with backoff. |
| `403` | `budget_exceeded` | Execute | No | Alias-based routing budget enforcement rejected the request. |
| `403` | `budget_threshold_missing` | Execute | No | A required routing budget threshold was missing. |
| `403` | `token_estimation_unavailable` | Execute | No | Keel could not estimate tokens needed for fail-closed routing. |
| `403` | `budget_estimation_unavailable` | Execute | No | Keel could not estimate cost needed for fail-closed routing. |
| `404` | `not_found` | Permits, request-level API | No | The requested resource does not exist for this project or route. |
| `409` | `conflict` | Permits, executions, execute, request-level API | No | The request conflicts with an existing resource or replay contract. |
| `409` | `idempotency_conflict` | Executions, execute, proxy, request-level API | No | The same idempotency key was reused with a different semantic request. |
| `500` | `internal_error` | Permits, executions, execute, proxy, request-level API | Yes | Keel failed internally before producing a more specific result. |
| `502` | `provider_error` | Executions, execute | Yes | The upstream provider request failed after permit approval. |
